Configuration Manager Web Frontend 2.0
Scott Keiffer, 2016 (eskeiffer@gmail.com)

Configuration Manager Web Frontend (CMFrontend) is a web application designed to provide quick access to information, tools, and deployments without the use of the Configuration Manager console. It is heavily inspired by Ron Crumbaker’s web tools for SMS 2003.

Update

If you already have the frontend installed all you will need to do is run update.ps1 via an elevated powershell session.

Change log

·         New users tab that allows you view and get quick access to all the computers that a username uses.

·         An OSD API has been included to help facilitate the creation of Pre-Execution applications / scripts. Please see the API documentation for more details.

·         The java applet has been replaced with a new C# launcher application & protocol that is browser agnostic. (Deployment required, see instructions below)

·         Remote Control has been added to the Tools list.

·         CMFrontend will now cleanup old and unused collections that are no longer needed.

·         Page load is now much faster as the task log is now limited to the last 25 entries.

·         Small style tweeks

Installation

Pre-requisites

1.       A New Server 2012 VM is Required. Do not attempt to install this on a site server.

a.       CMFrontend requires iis8 and .net 4.5 and will only work on server 2012.

b.       2 Gb of memory recomended

c.       Join to domain and install updates. The rest of the configuration will be done by the installer.

2.       Service accounts

a.       Domain access service account

                                                               i.      This account will be used to access AD groups to check for membership.

b.      ConfigMgr service account

                                                               i.      This account will be used to communicate with the ConfigMgr server. It must have adequate permissions.

                                                             ii.      See the service account installation section when setting up this account.

3.       AD Groups

a.       Frontend admin group

                                                               i.      All users in the group will have admin access to the site. It is recommended that this group only contain the users installing the frontend.

4.       Firewall access for WMI from the CMFrontend server to the Configuration Manager site server must be in place.

5.       Firewall access for SQL from the CMFrontend server to the Configuration Manager sql server must be in place.

6.       Firewall access for WMI from the CMFrontend server to all client systems must be inplace for machine policy update to work.

7.       This user running the installation must have full Configuration Manager access as well as the ability to add logins/users to the Configuration Manager SQL server.

Service Account Installation

The service account is used to perform all ConfigMgr & SQL actions. This will show you how to give a new service account the minimum amount of rights for CMFrontend to function. You may use an existing service account with greater access if you wish.

1.       Create the service account in AD

2.       Import the Frontend Service Account security role

a.       Open the ConfigMgr console

b.      Go to Administration > security

c.       Right click on Security Roles, select Import Security Role.

d.      Browse to the CMFrontend installation directory and select Frontend_Service_Account.xml

3.       Add the service account to administrative users

a.       Open the ConfigMgr console

b.      Go to Administration > security

c.       Right click on Administrative Users, select add user or group

d.      In the Add User or Group window, click browse and enter in your new service account

e.      Under Assigned security roles, hit add and select the newly imported Frontend Service Account role.

f.        Under Assigned security scopes and collections, select “All instances of the objects that are related to the assigned security roles”.

                                                               i.      If you do not select this, the account will not have the correct access.

4.       Check to see if the service account can connect by launching the console as the service account (Shift-Rightclick, run as different user).

 

Installation

The installation is mostly all automated, as long as the prerequisites are setup correctly this process should be seamless.

1.       Log into your new Server 2012 installation as an administrator that also has full access to ConfigMgr.

2.       Make sure the Server 2012 disk/iso is mounted.

3.       Open powershell as administrator.

4.       Run install.ps1

5.       After the install script is finished open a browser, navigate to the server you just installed.

6.       Login with a user that is in the AD frontend admin group.

7.       Select the collections each group should see and hit enable. (Tip: Only select the highest level limiting collections so you do not get duplicate entries)

8.       Repeat for each RBAC group as desired.

 

What does the installer do?

1.       Download and install SQL 2012 Express.

2.       Install requires IIS features.

3.       Installs the actual binary files.

4.       Configures all configuration files.

5.       Adds service account to the ConfigMgr database with read-only permissions.

6.       Adds service account to the Computer warranty database if present.

7.       Creates a Self-signed certificate.

8.       Configures IIS bindings.

9.       Installs the CMFrontend background service.

 

 

Helpdesk Launcher Deployment

The Helpdesk Launcher must be deployed to all system accessing the CMFrontend site if tools are to launch when clicked. You can find the install/uninstall scripts in the HelpdeskLauncher\Content folder of the installation files. For your convenience, I have also included a ConfigMgr export of the application that you can import into your environment. Details on importing the pre-built application below:

1.       Copy the contents of HelpdeskLauncher\ConfigMgr_Export to the file server you use to store application source files.

2.       In the ConfigMgr console, right click on applications and click import.

3.       In the Import application wizard, browse to the UNC path you placed the files and open the zip file.

4.       Next through the rest of the wizard to finish the import.

5.       Once the import is complete, deploy the application to a collection that contains the IT staff systems you wish this program to be installed on.

 

Known Issues

·         AD group checks currently use the group name to perform checks. Renaming AD groups after installation will cause login to stop working for users in that group.

·         If you deploy a program, all other programs for that package will not be deployable.

 

FAQ

A Collection is missing from the admin page for a group, how do I show that group?

This data is taken directly from the servers RABC records. It will only display collections explicitly defined in the RABC rules for that group. If you want to see a collection that is missing, you must add that collection the group’s collections in Administrative users.

 

When I deploy an application, where are the collection created?

All collections are created in the FrontEnd Software Distribution folder. With the following name: SoftDist - <RABC group> - Application name.

 

How often is inventory data gathered from the ConfigMgr server?

Inventory data is gathered for each computer roughly ever 24hrs. Each time data is received for a computer a random number between 18 and 30 hours is selected for the next refresh to prevent excessive load on the ConfigMgr SQL server.

 

Why does a deployment get stuck at “Waiting for client”?

At “Waiting for client” the server is checking to see if a deployment has finished. This means checking the status against the ConfigMgr SQL server. Depending on your client settings going from “Waiting for client” to “Deployment Complete” will be delayed until the data is current on the ConfigMgr SQL server.

 

Where are the log files stored?

There are two log files. One for IIS and another for the background service. Both are in the program data folder. C:\ProgramData\CMFrontEnd

 

How do I exclude an application from showing up in the applications list?

Assign the application the category: “FrontEndExclude”

 

How do I include packages/programs in the applications list?

Set the category for each program to “FrontEndInclude”

 

How do I change what information is displayed on the information tab?

You can edit the existing sql files or add new sql files in the following location: C:\ProgramData\CMFrontEnd\CompInfo. If you are adding a new sql file make sure you follow the name and query format are similar to the prepackaged sql files.

 

Does CMFrontend support Mac or Linux systems?

No.

 

Where is the API documentation?

The documentation lives on the webserver. Once cmfrontend is installed go to https://<server>/help to view the documentation.

 

What does CMFrontend remove when cleanup is run?

Every hour, deployment cleanup is run. When a deployment collection either has no members or the application that was being deploy is deleted or retired, CMfrontend will remove the following:

·         All internal database entries (including task log)

·         The collection in ConfigMgr that contain the deployment.

This process is retroactive as well. Once the 2.0 update is installed a mass cleanup will take place if you have had CMfrontend in production for a while.
